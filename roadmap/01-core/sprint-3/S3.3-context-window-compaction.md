# S3.3 — Context window & memory compaction

| Field | Value |
|-------|--------|
| **Task ID** | S3.3 |
| **Status** | Todo |

---

## Description

Design and implement a first-pass **context window management** strategy for the agent:
summarise or compact older conversation/tool outputs so that long runs and larger KB do not
cause uncontrolled growth in token usage (NF6). Category: **LLMOps tech debt** (context
window optimisation).

---

## Requirements

- [ ] Identify where conversation/history is accumulated for the agent (e.g. state passed
      between LangGraph nodes, previous tool outputs used as context).
- [ ] Define a simple policy for when to:
  - [ ] keep full detail,
  - [ ] replace detail with a compact summary,
  - [ ] drop non-essential history.
- [ ] Implement a compact/summarise function (can be LLM-based or heuristic) and wire it into
      the agent pipeline such that context is bounded for long runs.
- [ ] Ensure that NF6 limits (token budget, timeouts) have lower risk of being hit due purely
      to unbounded context growth.

---

## Checklist

- [ ] Add configuration knobs (e.g. `max_history_items`, `max_tokens_for_history`) in config
      or a dedicated settings module.
- [ ] Implement context compaction in a single place (e.g. a helper in `apps/agent/state` or a
      LangGraph callback) instead of scattered manual truncation.
- [ ] Document the trade-offs: what is summarised vs kept, and any known limitations.
- [ ] Optionally add a debug mode that logs when compaction happens and how much context was
      reduced.

---

## Test requirements

- [ ] A synthetic long-running scenario demonstrates that context length remains bounded
      (measured via token count or message count) and NF6 limits are not hit solely due to
      unbounded history.
- [ ] Normal short runs remain unaffected and continue to pass existing evals.
