## Description

Implement post-incident learning loop: postmortem template in `kb/postmortems/`, pipeline to re-index KB when postmortems change, and rule "each closed incident → add new eval case and run evals in CI". Goals: F9 (Should); roadmap Phase 5 / recurring.

---

## Requirements

- [ ] Postmortem template (e.g. `kb/postmortems/_template.md` or docs/runbooks/postmortem-template.md) with sections: summary, what we know, root cause, actions, lessons.
- [ ] Re-index pipeline: when new/changed file in `kb/postmortems/`, run embedding + pgvector index (script or webhook); document how to run.
- [ ] Rule: for each significant closed incident, add a postmortem and at least one new eval case (from P4.3 runbook); CI evals include it.
- [ ] Document in runbook or README: "After closing incident: 1) add postmortem, 2) re-index KB, 3) add eval case, 4) run CI."
- [ ] Optional: cron or GitHub Action to re-index on push to kb/postmortems/.

---

## Checklist

- [ ] Create postmortem template; document in runbook or kb/postmortems/README.
- [ ] Script or command to re-index KB (embed runbooks + postmortems, update pgvector); document in runbook.
- [ ] Document "post-incident checklist": add postmortem → re-index → add eval case (per P4.3) → push (CI runs evals).
- [ ] Optional: automate re-index on push to kb/ (e.g. GitHub Action).
- [ ] Link from roadmap 02-hardening README "Recurring" section.

---

## Test requirements

- Adding a new postmortem and running re-index script updates pgvector; agent can retrieve new postmortem.
- Following the checklist: new eval case runs in CI and is included in suite.
- Template is clear enough for an operator to fill after an incident.
