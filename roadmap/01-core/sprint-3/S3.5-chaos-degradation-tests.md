# S3.5 — Chaos / degradation test harness

| Field | Value |
|-------|--------|

---

## Description

Create a basic **chaos / degradation test harness** to simulate failures or slowdowns in
key dependencies (MCP servers, DB) and verify that the agent responds by escalating or
failing closed, not by hanging or acting unsafely. Category: **Resiliency tech debt**.

---

## Requirements

- [ ] Choose a simple tool or pattern (e.g. Toxiproxy, docker-compose profiles, or custom
      test fixtures) to introduce latency/errors into MCP and/or DB traffic.
- [ ] Define at least 2–3 chaos scenarios, e.g.:
  - [ ] Telemetry MCP very slow or returning 5xx,
  - [ ] KB MCP unavailable,
  - [ ] OPA unavailable or timing out.
- [ ] For each scenario, define an expected outcome (e.g. escalation packet with specific
      reason, no act/PR created).
- [ ] Integrate at least one scenario into automated tests or evals.

---

## Checklist

- [ ] Add scripts or configuration for running the system under degraded conditions
      (e.g. Toxiproxy docker service, or an alternative approach).
- [ ] Implement tests/evals that run with the degradation enabled and assert correct
      behaviour (escalation vs unsafe action).
- [ ] Document how to run chaos scenarios locally (step-by-step, including cleanup).
- [ ] Ensure logs/audit clearly indicate degraded conditions (helpful for debugging).

---

## Test requirements

- [ ] Running the chaos harness for each scenario produces the expected escalation/deny
      behaviour with no hangs.
- [ ] Tests/evals fail if the agent is modified to act unsafely under the same degraded
      conditions.
