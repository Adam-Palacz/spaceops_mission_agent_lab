# S3.2 — Prompt registry & versioning

| Field | Value |
|-------|--------|
| **Task ID** | S3.2 |
| **Status** | Done |

---

## Description

Introduce a **prompt registry** so core prompts (triage, investigate, decide, report, act)
are no longer only inline in code. Prompts should be externalised (e.g. to YAML/JSON or a
small config module), versioned, and referenced by ID from the agent code, with `prompt_id`
and `prompt_version` flowing into the LLM observability spine from S3.0. Category:
**LLMOps tech debt** (prompt lifecycle).

---

## Requirements

- [x] Inventory existing prompts used by the agent (at minimum: triage, investigate, decide,
      report, act).
- [x] Create a prompt registry representation (e.g. `prompts/*.yaml` or a dedicated Python
      module) with:
  - [x] stable IDs for each prompt,
  - [x] a place to record version/description,
  - [x] the prompt text itself.
- [x] Update agent nodes to load prompts via the registry, not hard-coded string literals
      scattered across the codebase.
- [x] Document how to change a prompt (including how to run evals/regressions afterwards).

---

## Checklist

- [x] Add a `prompts/` folder or equivalent with one or more files containing the core prompts,
      including IDs and optional metadata (e.g. `version`, `last_updated`, `owner`).
- [x] Refactor at least one key node (e.g. `decide`, `report`) to use the registry instead of
      inline prompt text, while preserving behaviour.
- [x] Ensure `evals` can reference prompts by ID if needed (optional hook for future work).
- [x] Add short docs (e.g. in `docs/` or README) explaining the prompt registry and how changes
      should always be accompanied by relevant eval runs.

---

## Test requirements

- [x] Agent runs still pass existing evals after moving prompts into the registry.
- [x] Changing a prompt in the registry (in a controlled way) is reflected in agent behaviour
      without code changes, and evals make regressions visible.
