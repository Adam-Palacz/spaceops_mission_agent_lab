# S3.4 — MCP/HTTP retry & circuit breaker layer

| Field | Value |
|-------|--------|
| **Task ID** | S3.4 |
| **Status** | Todo |

---

## Description

Introduce a shared **retry/backoff + circuit-breaker** layer for MCP and other HTTP calls
so that transient failures (network blips, 5xx) are handled gracefully and repeated errors
do not hammer downstream services. Category: **Resiliency tech debt**.

---

## Requirements

- [ ] Identify all places where MCP/HTTP calls are made (e.g. `apps/agent/mcp_client.py`,
      `apps/agent/opa_client.py`, API clients).
- [ ] Design a small helper or wrapper that supports:
  - [ ] configurable max retries,
  - [ ] exponential backoff with jitter,
  - [ ] basic circuit-breaker semantics (open/half-open/closed or a simplified version).
- [ ] Apply this helper to at least the most critical paths (e.g. MCP Telemetry, KB, Ticketing,
      GitOps, OPA), without changing their public interface.
- [ ] Ensure logging/audit captures when retries or circuit-breaker events occur.

---

## Checklist

- [ ] Add a module (e.g. `apps/common/http_resilience.py`) that exposes simple functions or
      context managers for resilient calls.
- [ ] Refactor MCP client and OPA client to use the new helper for outgoing HTTP connections.
- [ ] Document configuration knobs (e.g. max retries, backoff parameters) and defaults.
- [ ] Ensure that when the circuit breaker opens, the agent **fails closed** (escalation or
      clear error), not “best-effort” unsafe behaviour.

---

## Test requirements

- [ ] Unit tests (or focused integration tests) demonstrate:
  - [ ] retries happening on transient errors,
  - [ ] circuit breaker opening after repeated failures and short-circuiting further calls,
  - [ ] correct behaviour when services recover (optional half-open test).
- [ ] Existing evals still pass, and error paths now have clearer, more consistent behaviour
      under network/API failures.
