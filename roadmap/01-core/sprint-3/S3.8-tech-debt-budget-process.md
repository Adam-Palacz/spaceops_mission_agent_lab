# S3.8 — Tech-debt budget process & documentation

| Field | Value |
|-------|--------|
| **Task ID** | S3.8 |
| **Status** | Done |

---

## Description

Codify a **tech-debt budget process** so that roughly 20% of sprint capacity after S2 is
reserved for refactors, dead-code removal, readability, and other non-feature work. Category:
**Process tech debt**.

---

## Requirements

- [x] Define, in prose, what “tech debt work” means for this project (examples + non-examples).
- [x] Propose a simple rule (e.g. “~20% of story points / tasks per sprint are tech-debt items”)
      and how it is tracked (e.g. tags/labels in issues or BOARD rows).
- [x] Update documentation (e.g. `roadmap/base-roadmap.md`, `README.md`, or a dedicated
      `docs/process.md`) to describe the policy.
- [x] Optionally, add light-weight metrics or checklists to ensure the budget is actually used
      (e.g. per-sprint review section for debt items completed).

---

## Checklist

- [x] Write a short process doc describing:
  - [x] how to propose a tech-debt task,
  - [x] how it is prioritised against features,
  - [x] how to avoid “stealth features” masquerading as debt.
- [x] Add a note or section to the main README or roadmap so new contributors see the policy.
- [x] Ensure Sprint 3 tasks are tagged or grouped as “debt” so they can be used as examples.
- [x] Add a template snippet (e.g. in BOARDs or issue templates) reminding maintainers to
      include tech debt items when planning a sprint.

---

## Test requirements

- [x] The process is clear enough that an external reviewer could verify, for a given sprint,
      whether the ~20% tech-debt rule was respected.
- [x] Future sprint planning docs can reference this policy without ambiguity.
