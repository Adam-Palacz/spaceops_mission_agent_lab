# S3.4 — MCP/HTTP retry & circuit breaker layer

| Field | Value |
|-------|--------|
| **Task ID** | S3.4 |
| **Status** | Done |

---

## Description

Introduce a shared **retry/backoff + circuit-breaker** layer for MCP and other HTTP calls
so that transient failures (network blips, 5xx) are handled gracefully and repeated errors
do not hammer downstream services. Category: **Resiliency tech debt**.

---

## Requirements

- [x] Identify all places where MCP/HTTP calls are made (e.g. `apps/agent/mcp_client.py`,
      `apps/agent/opa_client.py`, API clients).
- [x] Design a small helper or wrapper that supports:
  - [x] configurable max retries,
  - [x] exponential backoff with jitter,
  - [x] basic circuit-breaker semantics (open/half-open/closed or a simplified version).
- [x] Apply this helper to at least the most critical paths (e.g. MCP Telemetry, KB, Ticketing,
      GitOps, OPA), without changing their public interface.
- [x] Ensure logging/audit captures when retries or circuit-breaker events occur.

---

## Checklist

- [x] Add a module (e.g. `apps/common/http_resilience.py`) that exposes simple functions or
      context managers for resilient calls.
- [x] Refactor MCP client and OPA client to use the new helper for outgoing HTTP connections.
- [x] Document configuration knobs (e.g. max retries, backoff parameters) and defaults.
- [x] Ensure that when the circuit breaker opens, the agent **fails closed** (escalation or
      clear error), not “best-effort” unsafe behaviour.

---

## Test requirements

- [x] Unit tests (or focused integration tests) demonstrate:
  - [x] retries happening on transient errors,
  - [x] circuit breaker opening after repeated failures and short-circuiting further calls,
  - [x] correct behaviour when services recover (optional half-open test).
- [x] Existing evals still pass, and error paths now have clearer, more consistent behaviour
      under network/API failures.
