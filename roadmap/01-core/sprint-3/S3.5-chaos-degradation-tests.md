# S3.5 — Chaos / degradation test harness

| Field | Value |
|-------|--------|
| **Task ID** | S3.5 |
| **Status** | Done |

---

## Description

Create a basic **chaos / degradation test harness** to simulate failures or slowdowns in
key dependencies (MCP servers, DB) and verify that the agent responds by escalating or
failing closed, not by hanging or acting unsafely. Category: **Resiliency tech debt**.

---

## Requirements

- [x] Choose a simple tool or pattern (e.g. Toxiproxy, docker-compose profiles, or custom
      test fixtures) to introduce latency/errors into MCP and/or DB traffic.
- [x] Define at least 2–3 chaos scenarios, e.g.:
  - [x] Telemetry MCP very slow or returning 5xx,
  - [x] KB MCP unavailable,
  - [x] OPA unavailable or timing out.
- [x] For each scenario, define an expected outcome (e.g. escalation packet with specific
      reason, no act/PR created).
- [x] Integrate at least one scenario into automated tests or evals.

---

## Checklist

- [x] Add scripts or configuration for running the system under degraded conditions
      (e.g. Toxiproxy docker service, or an alternative approach).
- [x] Implement tests/evals that run with the degradation enabled and assert correct
      behaviour (escalation vs unsafe action).
- [x] Document how to run chaos scenarios locally (step-by-step, including cleanup).
- [x] Ensure logs/audit clearly indicate degraded conditions (helpful for debugging).

---

## Test requirements

- [x] Running the chaos harness for each scenario produces the expected escalation/deny
      behaviour with no hangs.
- [x] Tests/evals fail if the agent is modified to act unsafely under the same degraded
      conditions.
