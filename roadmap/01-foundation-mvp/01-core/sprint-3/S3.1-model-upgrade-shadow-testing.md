## Description

Design and implement a minimal **model upgrade / shadow-testing pipeline** so that when the
current LLM provider deprecates or replaces a model, SpaceOps can safely compare current vs
candidate models **offline** before switching. In Sprint 3 this means:

- adding a central place to declare the current model and candidate model IDs,
- providing a script that runs existing evals against both and writes a comparison report,
- keeping the normal agent path on the current model (no automatic switch yet).

Building a full production rollout (scheduled runs, dashboards, formal switch criteria) is
deferred to Phase 4 hardening (see P4.8). Category: **LLMOps tech debt** (model lifecycle).

---

## Requirements

- [x] Single configuration point (e.g. in `config.py` or a small gateway module) describing:
  - [x] current production model identifiers (for triage/decide/report),
  - [x] one or more candidate models for shadow testing.
- [x] Ability to run a **shadow eval**: run the existing eval suite (or a subset) against both
      current and candidate models, capturing scores separately.
- [x] Clear output/report (CLI or file) that highlights:
  - [x] cases where the candidate model regresses vs current,
  - [x] overall summary (pass/fail) based on thresholds.
- [x] No change to default behaviour: normal runs still use the current model unless a
      dedicated flag/config is set.

---

## Checklist

- [x] Add a small abstraction around `_chat_completion` / model selection so that the model
      used for triage/decide/report can be swapped via config and/or env.
- [x] Implement a script or mode (e.g. `python -m evals.shadow_models`) that:
  - [x] runs a subset of evals using the current model,
  - [x] runs the same subset using a candidate model,
  - [x] writes a comparison report (JSON/markdown) under `evals/reports/`.
- [x] Document how to configure candidate models and how to interpret the shadow report.
- [x] Ensure that if the candidate model misbehaves (timeouts/errors), the script fails
      clearly but does **not** affect normal agent runs.

---

## Test requirements

- [x] Running the shadow-testing script with a dummy “candidate model” configuration produces
      a report file and does not affect normal eval runs.
- [x] When the candidate model is intentionally degraded (e.g. forced wrong outputs in tests),
      the report clearly surfaces regressions and returns a non-zero exit code.
