# S3.1 — Model upgrade / shadow-testing pipeline

| Field | Value |
|-------|--------|

---

## Description

Design and implement a minimal **model upgrade / shadow-testing pipeline** so that when the
current LLM provider deprecates or replaces a model, SpaceOps can safely migrate without
breaking triage/decide/report semantics. This should run selected evals against both the
current and candidate models and surface regressions before a switch, building on the LLM
observability spine from S3.0 for per-model metrics and comparisons. Category: **LLMOps
tech debt** (model lifecycle).

---

## Requirements

- [ ] Single configuration point (e.g. in `config.py` or a small gateway module) describing:
  - [ ] current production model identifiers (for triage/decide/report),
  - [ ] one or more candidate models for shadow testing.
- [ ] Ability to run a **shadow eval**: run the existing eval suite (or a subset) against both
      current and candidate models, capturing scores separately.
- [ ] Clear output/report (CLI or file) that highlights:
  - [ ] cases where the candidate model regresses vs current,
  - [ ] overall summary (pass/fail) based on thresholds.
- [ ] No change to default behaviour: normal runs still use the current model unless a
      dedicated flag/config is set.

---

## Checklist

- [ ] Add a small abstraction around `_chat_completion` / model selection so that the model
      used for triage/decide/report can be swapped via config and/or env.
- [ ] Implement a script or mode (e.g. `python -m evals.shadow_models`) that:
  - [ ] runs a subset of evals using the current model,
  - [ ] runs the same subset using a candidate model,
  - [ ] writes a comparison report (JSON/markdown) under `evals/reports/`.
- [ ] Document how to configure candidate models and how to interpret the shadow report.
- [ ] Ensure that if the candidate model misbehaves (timeouts/errors), the script fails
      clearly but does **not** affect normal agent runs.

---

## Test requirements

- [ ] Running the shadow-testing script with a dummy “candidate model” configuration produces
      a report file and does not affect normal eval runs.
- [ ] When the candidate model is intentionally degraded (e.g. forced wrong outputs in tests),
      the report clearly surfaces regressions and returns a non-zero exit code.
