# P4.8 — Model upgrade / shadow-testing rollout

| Field | Value |
|-------|--------|
| **Task ID** | P4.8 |
| **Status** | Todo |

---

## Description

Take the existing **shadow-testing pipeline** (S3.1) and turn it into a first-class part of
the production lifecycle: scheduled runs, dashboards, and a documented process for deciding
when to switch the primary model.

Category: **Hardening / LLMOps** (model lifecycle in production).

---

## Requirements

- [ ] Wire `python -m evals.shadow_models` into a scheduled or on-demand CI job that can be run
      before model changes.
- [ ] Persist and expose shadow reports (e.g. `evals/reports/` + link in docs or dashboard),
      so humans can compare models over time.
- [ ] Define simple decision rules for switching models (e.g. “no regressions on standard
      cases, 0 unsafe injection cases, acceptable latency/cost deltas”).
- [ ] Document the process in `docs/` (when to run shadow evals, how to read the report,
      who decides on model changes).

---

## Checklist

- [ ] Add a CI job (or GitHub Actions workflow) that can run `python -m evals.shadow_models`
      on demand or on a schedule.
- [ ] Ensure shadow runs do not affect normal evals or production traffic; failures only gate
      model changes.
- [ ] Add documentation page (e.g. `docs/shadow_models.md`) that links to sample reports and
      explains interpretation.
- [ ] Reference this process from `02-production-scale.md` and/or `docs/process.md` so model
      upgrade decisions follow a repeatable pattern.

---

## Test requirements

- [ ] Triggering the shadow-testing CI job produces a report, sets an appropriate success/fail
      status, and does not impact normal evals or the agent’s behaviour.
- [ ] An external reviewer can look at the report + docs and understand whether a candidate
      model is safe to promote.
