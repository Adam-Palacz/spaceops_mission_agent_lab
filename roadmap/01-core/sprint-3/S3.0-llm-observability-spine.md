# S3.0 — LLM observability spine (Langfuse-compatible)

| Field | Value |
|-------|--------|
| **Task ID** | S3.0 |
| **Status** | Done |

---

## Description

Introduce a lightweight **LLM observability spine** so that all LLM calls (triage, investigate,
decide, report, act, evals) are consistently logged with run and experiment metadata, and can
later be wired to a Langfuse-style backend. This is not a full Langfuse deployment; it is a
minimal, code-level API and data model (run → traces → LLM calls) that S3.1 (shadow-testing)
and S3.2 (prompt registry) can build on. Category: **LLMOps tech debt** (observability).

---

## Requirements

- [x] Define a small, central **LLM observability API** (e.g. `apps/telemetry_llm.py` or similar)
      that can:
  - [x] create / attach to a logical `run_id` (aligned with existing trace_id / audit where possible),
  - [x] record each LLM call with at least: `run_id`, `node` (triage/investigate/decide/report/act),
        `model_id`, `prompt_id`, `prompt_version`, optional `eval_case_id` / `injection_case_id`,
        timing and token usage when available,
  - [x] emit this data to one or more sinks (OTel spans, structured logs, simple NDJSON/DB table).
- [x] Keep the abstraction **Langfuse-compatible** in spirit:
  - [x] concepts map cleanly to run / trace / span and model/prompt metadata,
  - [x] backends can later include Langfuse without changing call sites in agent code.
- [x] No new external dependency is required in S3.0; use existing OTel / logging / JSON storage.
- [x] Default behaviour: when the observability spine is disabled or misconfigured, agent still
      runs; logging should degrade gracefully (no hard failures).

---

## Checklist

- [x] Add a module (e.g. `apps/llm_observability.py` or `apps/telemetry_llm.py`) that exposes
      functions such as:
  - [x] `start_llm_run(run_id: str | None, **metadata) -> run_id`,
  - [x] `log_llm_call(run_id, node, model_id, prompt_id, prompt_version, tags=None, metrics=None)`.
- [x] Wire this module into core LLM call sites (triage, investigate, decide, report, act) and,
      where reasonable, evals (`evals/scoring.py`) so that every call can be attributed to a
      model and prompt version.
- [x] Ensure `model_id` and `prompt_id` / `prompt_version` are always provided in one place
      (S3.1/S3.2 will define how they are chosen); for now, stub them from existing config and
      inline prompts.
- [x] Implement at least one sink:
  - [x] OTel span attributes for LLM calls (reusing existing tracing),
  - [x] AND/OR append-only NDJSON under `data/llm_runs/` for quick inspection.
- [x] Document the data model and how it relates to future Langfuse-style monitoring (e.g. in
      `docs/llm_observability.md` or a short section in the roadmap).

---

## Test requirements

- [x] A simple test or script can run a few agent calls and show:
  - [x] consistent `run_id` across LLM calls in a run,
  - [x] logged `model_id` and (stubbed) `prompt_id` / `prompt_version` for each call.
- [x] Existing evals (`python -m evals.scoring`) can be configured to tag runs with
      `eval_case_id` / `injection_case_id` in the observability spine without changing
      their core logic.
- [x] Disabling or misconfiguring the spine does **not** break normal agent or API runs;
      it only reduces observability.
