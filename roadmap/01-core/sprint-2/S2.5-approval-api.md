# S2.5 — Approval API (idempotent, auth)

| Field | Value |

---

## Description

Implement approval API: `GET /approvals`, `POST /approvals/:id/approve`, `POST /approvals/:id/reject`. Idempotent (repeated approve same id = same result); require authentication (API key or token in MVP); record who approved/rejected and when in audit (NF9). Goals: F8, NF9.

---

## Requirements

- [ ] `GET /approvals` returns list of pending (and optionally all) approval requests.
- [ ] `POST /approvals/:id/approve` and `POST /approvals/:id/reject` update request status; require auth (e.g. API key in header or Bearer token).
- [ ] Idempotent: calling approve again for already-approved id returns success and does not re-execute (S2.6).
- [ ] Audit: each approve/reject writes to audit log with actor=human, who (identity from auth), when, decision, outcome.
- [ ] Storage: approval_requests table or file; status = pending | approved | rejected.

---

## Checklist

- [ ] Add DB table or file for approval_requests (id, incident_id, action payload, status, created_at, decided_at, decided_by).
- [ ] Implement GET /approvals (filter by status if needed).
- [ ] Implement POST /approvals/:id/approve and reject; check auth; update status; write audit entry (actor=human, who, when).
- [ ] Idempotent: if status already approved/rejected, return 200 and do not change.
- [ ] Document auth (header name, env for API key).

---

## Test requirements

- GET /approvals returns pending requests; approve/reject updates status.
- Unauthenticated request to approve/reject returns 401.
- Second approve on same id returns 200 and does not trigger execution again (S2.6).
- Audit log contains entry with actor=human and timestamp for each decision.
