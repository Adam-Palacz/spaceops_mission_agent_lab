# S2.6 — Execute restricted action on approve

| Field | Value |
|-------|--------|
| **Task ID** | S2.6 |
| **Status** | Todo |

---

## Description

When an approval request is approved via API (S2.5), execute the restricted action (e.g. call GitOps MCP to create PR); write result to audit log. Execution happens only once per approval (idempotent approve in S2.5 prevents double execution).

---

## Requirements

- [ ] On `POST /approvals/:id/approve`: load request, if status was pending execute the stored action (e.g. GitOps MCP with stored params), update status to approved, write outcome to audit log.
- [ ] Execution is one-shot per approval id; if already approved, do not execute again.
- [ ] Audit log: entry with actor=agent (or "system"), tool, args_hash, decision=allow, outcome=success|failure.
- [ ] On execution failure: record failure in audit; optionally mark request as failed or leave approved with outcome=failure.

---

## Checklist

- [ ] In approval API handler for approve: after updating status, call execution layer with stored action (tool + args).
- [ ] Execution layer: invoke corresponding MCP (e.g. GitOps create_pr); capture result; write audit entry.
- [ ] Guard: if request already approved/rejected, skip execution (idempotent).
- [ ] Handle MCP errors: write outcome=failure to audit; return 200 to client with message or store failure state.

---

## Test requirements

- Approving a pending request triggers execution once; audit shows outcome.
- Approving the same id again does not execute again; audit or logs show single execution.
- Failed execution (e.g. GitOps error) is recorded in audit; client gets clear response.
