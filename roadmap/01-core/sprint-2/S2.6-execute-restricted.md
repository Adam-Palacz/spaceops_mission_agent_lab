# S2.6 — Execute restricted action on approve

| Field | Value |
|-------|--------|
| **Task ID** | S2.6 |
| **Status** | Done |

---

## Description

When an approval request is approved via API (S2.5), execute the restricted action (e.g. call GitOps MCP to create PR); write result to audit log. Execution happens only once per approval (idempotent approve in S2.5 prevents double execution).

---

## Requirements

- [x] On `POST /approvals/:id/approve`: load request, if status was pending execute the stored action (e.g. GitOps MCP with stored params), update status to approved, write outcome to audit log.
- [x] Execution is one-shot per approval id; if already approved, do not execute again.
- [x] Audit log: entry with actor=agent (or "system"), tool, args_hash, decision=allow, outcome=success|failure.
- [x] On execution failure: record failure in audit; optionally mark request as failed or leave approved with outcome=failure.

---

## Checklist

- [x] In approval API handler for approve: after updating status, call execution layer with stored action (tool + args).
- [x] Execution layer: invoke corresponding MCP (e.g. GitOps create_pr); capture result; write audit entry.
- [x] Guard: if request already approved/rejected, skip execution (idempotent).
- [x] Handle MCP errors: write outcome=failure to audit; return 200 to client with message or store failure state.

---

## Test requirements

- [x] Approving a pending request triggers execution once; audit shows outcome.
- [x] Approving the same id again does not execute again; audit or logs show single execution.
- [x] Failed execution (e.g. GitOps error) is recorded in audit; client gets clear response.

---

## Implementation notes

- **Execution layer**: `apps/agent/approval_executor.py` — `execute_approved_action(approval_id, rec)` maps `action_type` to MCP or no-op: `change_config` → GitOps `create_pr` (branch `agent/{incident_id}-approved-{id}`); `restart_service` → no-op in MVP (audit only).
- **API**: `apps/api/main.py` — approve handler loads request with `get_request` before `store_approve`; runs executor only when `rec_before["status"] == "pending"`; appends audit entry with `actor=agent`, `tool=execute_restricted`, `outcome` and optional `error_message`; response includes `execution: { outcome, result, error_message }` when execution ran.
- **Tests**: `tests/test_api.py` — `test_approvals_approve_reject_flow_and_idempotent` (mock executor, assert first approve has `execution`, second has no `execution`, single `execute_restricted` in audit); `test_approvals_approve_execution_failure_recorded` (mock failure, assert response and audit show failure).
