# S1.18 — Audit log: outcome failure vs empty, error details

| Field | Value |
|-------|--------|
| **Task ID** | S1.18 |
| **Status** | Todo |

---

## Description

In Investigate, every MCP call is audited with `outcome="success"`. The MCP client catches exceptions and returns `[]`, so the audit log does not distinguish "tool returned 0 results" from "tool failed (exception)". This task adds explicit outcomes and optional error info so audits are accurate for debugging and compliance.

---

## Requirements

- [ ] **Outcome semantics:** Support at least `success` (tool ran and may have returned data), `empty` (tool ran, returned no results), `failure` (tool threw or returned error). Log `outcome` accordingly after each MCP call in the agent.
- [ ] **On failure:** Log a short, safe error indication in the audit entry (e.g. `error_message` or `error_hash` + brief message), without leaking full stack traces or PII. Prefer hash + short code/message for consistency.
- [ ] **Backward compatibility:** Existing readers of the audit log should still see `outcome`; new optional fields (e.g. `error_message`) can be absent for success/empty.

---

## Checklist

- [ ] Change MCP client (or its call sites) to return a small result type that indicates success / empty / failure (e.g. tuple `(list, outcome, error_message)` or a tiny dataclass). Alternatively, in the agent layer catch exceptions from MCP and pass outcome + error to `audit_append`.
- [ ] In `audit_log.append_entry`: accept optional `error_message` (or `error_hash`); extend schema doc to include outcome values and optional error field.
- [ ] In `nodes.investigate`: for each MCP call, set outcome to `success` | `empty` | `failure` and pass through any error message; call `audit_append(..., outcome=..., error_message=...)` as needed.
- [ ] Tests: audit entries show `failure` when MCP raises; show `empty` when MCP returns []; show `success` when MCP returns non-empty (and optionally check error_message absent).

---

## Test requirements

- [ ] Test: audit entry for MCP tool has `outcome` in {success, empty, failure}; assert in tests/test_audit_log.py (or equivalent).
- [ ] Test: when MCP call raises (mocked), audit entry has `outcome=failure` and optional error field (e.g. error_message or error_hash) set; no stack trace or PII.
- [ ] Test: when MCP call returns [] (no exception), audit entry has `outcome=empty` (not success).
- [ ] Test: when MCP call returns non-empty list, audit entry has `outcome=success`; error field absent.
- [ ] Existing audit log tests still pass; schema backward compatible.
