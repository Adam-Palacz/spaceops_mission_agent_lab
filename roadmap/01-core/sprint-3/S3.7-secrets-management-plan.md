# S3.7 — Secrets management plan & integration path

| Field | Value |
|-------|--------|
| **Task ID** | S3.7 |
| **Status** | Done |

---

## Description

Define and start implementing a **secrets management plan** that moves the system away from
long-term reliance on `.env` files towards a proper secrets backend (e.g. Vault or a cloud
secrets manager). Category: **Infra/Sec tech debt**.

---

## Requirements

- [x] Inventory all secrets used by the system (OpenAI keys, DB passwords, MCP tokens,
      GitHub tokens, etc.).
- [x] Choose a target secrets backend for non-local environments (e.g. HashiCorp Vault,
      AWS Secrets Manager, GCP Secret Manager).
- [x] Define an abstraction layer in code (e.g. a small `secrets` module or integration
      points in `config.py`) that can fetch secrets from the backend while still allowing
      `.env` for local development.
- [x] Document a migration path: local → staging → production, including rotation.

---

## Checklist

- [x] Add documentation (e.g. `docs/secrets.md`) describing the chosen backend, how secrets
      are named, and how they map to current env vars.
- [x] Implement a minimal integration stub (even if mocked) that shows how the agent/API
      would read from the secrets backend instead of `.env`.
- [x] Ensure that local development remains simple (e.g. `.env` still supported) while
      making it clear it is **not** the long-term production mechanism.
- [x] Identify at least one high-value secret (e.g. `OPENAI_API_KEY`) to be the first to
      move under the new mechanism in a future sprint.

---

## Test requirements

- [x] A small demo or test proves that replacing `.env` with a real secrets backend (or
      its mock) does not break the application startup.
- [x] Documentation is clear enough that a new engineer can understand how secrets will
      be managed and rotated once the backend is wired in.
